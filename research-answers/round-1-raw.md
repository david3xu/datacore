Q1 — How Delta Lake stores data on disk (concrete, file-level)
- What Delta writes
  - Data files: parquet files containing the columnar row data (one or more files under the table path).
  - Transaction log: _delta_log/ directory containing an append-only sequence of JSON (and later parquet checkpoint) files that record transactions (add/remove file entries, protocol, metadata).
  - Checkpoint files: periodically a compacted Parquet checkpoint file is written under _delta_log/ (named like 00000000000000000010.checkpoint.parquet) so readers can avoid replaying all JSON logs.
  - _delta_log/ also contains small _txn or commit marker files used by some writers (implementation detail) and a _last_checkpoint hint file when checkpoints exist.

- Example: create a Delta table and insert 100 rows
  - After creating an empty Delta table you will have:
    - /path/to/table/_delta_log/00000000000000000000.json (initial metadata transaction)
    - maybe 00000000000000000000.checkpoint.parquet (if a checkpoint was forced)
  - After inserting 100 rows (single append write) you will have:
    - one or more parquet data files under /path/to/table/ (names autogenerated, e.g. part-00000-... .parquet). The write may produce a single parquet file or multiple files depending on shuffle/partitions and target file size.
    - new entry in _delta_log/, e.g. /path/to/table/_delta_log/00000000000000000001.json containing the AddFile entries for the newly written parquet file(s) plus an operation record describing the append.
    - Over time, additional numbered JSON log files are created for each transaction. Periodically a checkpoint parquet is written (e.g., 00000000000000000050.checkpoint.parquet) and older JSON entries can be removed by vacuum/retention/compaction processes.

- Minimal concrete file-tree example (after create + single append of 100 rows)
  - /mnt/delta/events/
    - part-00000-aaaa-bbbb-cccc.parquet      ← actual data (maybe only one file)
    - _delta_log/
      - 00000000000000000000.json            ← create table / metadata
      - 00000000000000000001.json            ← append transaction (AddFile records for part-00000)
      - 00000000000000000001.checkpoint.parquet (only present if a checkpoint was written; often appears later)

- Example content of a transaction JSON (simplified)
  - 00000000000000000001.json will contain entries like:
    - { "add": { "path":"part-00000-aaaa-bbbb-cccc.parquet", "size": 12345, "partitionValues": {...}, "modificationTime": 1_680_000_000_000 } }
    - { "metaData": {...} } (only in metadata transaction)
    - { "commitInfo": { "timestamp": ... , "operation": "WRITE", "operationParameters": {"mode":"Append"} } }

(These are exactly the files Delta maintains — parquet data files + the _delta_log JSON/Parquet checkpoint records. See Databricks Delta Lake storage & transaction log docs for the exact schema and examples.)

Q2 — Difference: ADLS Gen2 vs Cosmos DB vs Azure SQL (and concrete choice for 10k events/day)
- Short, concrete differentiation (when to use each)
  1) Azure Data Lake Storage Gen2 (ADLS Gen2)
     - Nature: object/blob storage with hierarchical namespace, optimized for big-data workloads, large files, analytics.
     - Use when: append-heavy event stores, raw/landing (Bronze) data, large-scale ETL, parquet/columnar files for analytics, cost-efficient for TBs of data.
     - Typical access: batch reads (Spark/Databricks), POSIX-like directory layout, mountable in Databricks.
     - Example role: Bronze and Silver Delta Lake tables live on ADLS Gen2.
  2) Azure Cosmos DB
     - Nature: globally distributed multi-model database (document/key-value), low-latency single-digit-ms reads/writes, offers per-item indexing and point lookups.
     - Use when: OLTP-style workloads, per-event real-time serving, storing JSON documents for low-latency queries at scale, multi-region replication, high RPS with low latency.
     - Example role: real-time serving layer or event store for microservices; not ideal as a primary analytics store for large parquet scans.
  3) Azure SQL (Managed Instance / Azure SQL Database)
     - Nature: relational OLTP DB with ACID transactions, SQL engine, good for structured relational workloads and BI that needs transactional consistency and complex joins at query time.
     - Use when: transactional relational data, serving dashboards requiring relational semantics, business apps expecting T-SQL and row-level updates.
     - Example role: Gold (curated) relational tables for BI/reporting served to downstream apps.

- Concrete example for 10,000 event records/day (recommended layers)
  - Bronze (raw landing): ADLS Gen2 (Delta Lake)
    - Why: cheap per-GB storage, efficient for append, integrates with Databricks Spark for transformation, stores data as partitioned parquet via Delta.
    - Concrete: write daily partitioned Delta table on ADLS Gen2 at path: abfss://raw@<account>.dfs.core.windows.net/events/date=2026-03-20/*.parquet
  - Silver (cleaned/enriched): ADLS Gen2 (Delta Lake)
    - Why: do cleaning, dedupe, parse fields, do light enrichment in Spark; keep as Delta because it supports ACID over parquet and time travel for audit.
  - Gold (serving / BI / low-latency)
    - Option A (analytics / dashboards): Azure SQL Managed Instance or Synapse SQL Pool if BI tools need relational endpoints (frequent joins, small tables).
      - Use-case: aggregated daily metrics, slowly changing dimensions—store final aggregates in Azure SQL for dashboard refreshes.
    - Option B (low-latency per-item reads / real-time querying): Azure Cosmos DB
      - Use-case: if dashboards or APIs need to look up individual events or entity state with <10ms latency, or global distribution is required.
  - Concrete mapping for 10k/day:
    - Bronze: ADLS Gen2 Delta table, partitioned by ingestion_date (very cheap and performant for scans).
    - Silver: ADLS Gen2 Delta with cleaned columns and appropriate partitioning (e.g., date or event_type).
    - Gold: If your consumers require transactional row-level updates and relational queries → push aggregates/lookup tables into Azure SQL; if they require sub-second document lookups globally → push relevant items into Cosmos DB.

- Example cost/perf rationale
  - 10k events/day ≈ 3.65M events/year — easily stored on ADLS Gen2 as parquet (low cost).
  - Use Cosmo DB only if you have strong low-latency serving or multi-region replication requirements; otherwise Gold can be relational (Azure SQL) or simply a Delta table exposed via serverless SQL or Synapse for BI.

Q3 — How Delta Lake handles schema evolution (concrete commands & examples)
- Two common patterns:
  - 1) Controlled schema change with SQL ALTER TABLE ADD COLUMNS
  - 2) Schema merge on writes (allowing writers to append data that contains new columns) using write options (mergeSchema=true in DataFrame writes) or use of MERGE INTO with schema evolution options

- Initial table
  - Suppose we create a Delta table with columns (id, timestamp, source, content):

SQL (Databricks/Delta SQL):
  CREATE TABLE delta_db.events (
    id STRING,
    timestamp TIMESTAMP,
    source STRING,
    content STRING
  )
  USING DELTA
  LOCATION 'abfss://<container>@<account>.dfs.core.windows.net/delta/events';

PySpark example (create from DataFrame):
  from pyspark.sql import SparkSession
  spark = SparkSession.builder.getOrCreate()
  df = spark.createDataFrame([
      ("1", "2026-03-20T00:00:00", "svcA", "hello")
  ], schema=["id","timestamp","source","content"])
  df.write.format("delta").mode("overwrite").save("/mnt/delta/events")

- Add new columns (actor, event_type) — three concrete approaches

Approach A — ALTER TABLE to add columns (explicit)
  -- SQL
  ALTER TABLE delta_db.events
  ADD COLUMNS (
    actor STRING,
    event_type STRING
  );

  -- After this, the table schema is updated in the latest _delta_log metadata transaction. Existing parquet files remain unchanged (they simply lack these columns). Reads will return NULL for the new columns for older files.

Approach B — Schema merge on write (automatic merge of writer schema into table schema)
  - Use when your incoming DataFrame has the additional columns and you want Delta to evolve the table schema as part of the append.

PySpark example with mergeSchema:
  from pyspark.sql.types import StructType, StructField, StringType, TimestampType
  new_schema = StructType([
    StructField("id", StringType(), True),
    StructField("timestamp", TimestampType(), True),
    StructField("source", StringType(), True),
    StructField("content", StringType(), True),
    StructField("actor", StringType(), True),
    StructField("event_type", StringType(), True)
  ])
  df_new = spark.createDataFrame([
    ("2", "2026-03-20T01:00:00", "svcB", "world", "alice", "click")
  ], schema=new_schema)

  # Append with schema merge enabled
  df_new.write \
        .format("delta") \
        .mode("append") \
        .option("mergeSchema", "true") \
        .save("/mnt/delta/events")

  # OR when using the table API:
  df_new.write \
        .format("delta") \
        .mode("append") \
        .option("mergeSchema", "true") \
        .saveAsTable("delta_db.events")

  What happens:
  - Delta will write a new transaction JSON in _delta_log/ that contains the metadata change merging the new fields into the table schema (equivalent to ALTER TABLE). Existing data files are not touched; they simply do not contain the new column (so read returns NULL).
  - The transaction log records the schema change in the metadata entry, and future readers will see the new schema.

Approach C — MERGE INTO with schema evolution (for upserts)
  - Use MERGE INTO with options to allow schema evolution (Databricks SQL or PySpark).
SQL example:
  MERGE INTO delta_db.events AS target
  USING (SELECT '3' AS id, CAST('2026-03-20' AS TIMESTAMP) AS timestamp,
                'svcC' AS source, 'payload' AS content,
                'bob' AS actor, 'purchase' AS event_type) AS source
  ON target.id = source.id
  WHEN MATCHED THEN UPDATE SET *
  WHEN NOT MATCHED THEN INSERT *

PySpark with merge and schema evolution (Databricks runtime supports schema evolution in MERGE):
  from delta.tables import DeltaTable
  deltaTable = DeltaTable.forPath(spark, "/mnt/delta/events")
  sourceDF = spark.createDataFrame([...], schema=new_schema)
  deltaTable.alias("t").merge(
      sourceDF.alias("s"),
      "t.id = s.id"
  ).whenMatchedUpdateAll().whenNotMatchedInsertAll().execute()

  # To enable automatic schema evolution in some runtimes, you may need:
  spark.conf.set("spark.databricks.delta.schema.autoMerge.enabled", "true")
  # Then the merge may update schema automatically when needed.

- What really happens on disk & in the log when schema evolves
  - A metadata JSON entry is written into _delta_log/ (next transaction file) that contains the new schema (the table metadata is versioned).
  - Existing parquet files are unchanged (they don't contain the new columns). Delta's read layer merges metadata schema with the file schema to present a unified view; missing columns show as NULL.
  - No immediate rewrite of existing data is required. If you need to compact or physically add columns with defaults, you can rewrite files with a CTAS or write overwrite with present schema.

- Backward/forward compatibility notes
  - Adding nullable columns is safe (old rows will return NULL).
  - Adding non-nullable columns with no default will break reads/writes; recommended to add as nullable and then later backfill and change constraints if necessary.
  - Renaming/dropping columns is supported via ALTER TABLE or programmatic rewrite but may require rewriting or using explicit rename operations supported by Delta (Databricks docs show ALTER TABLE ... RENAME COLUMN and DROP COLUMN support depending on Delta version).
